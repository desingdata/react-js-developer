01

Middlewares: Concepto y Creación

¿Qué es un middleware?

Un middleware en Redux es una función que se sitúa entre el dispatch de una acción y el reducer. Puede inspeccionar, modificar, retardar o incluso cancelar acciones. Es la base para asincronía y logging.

dispatch(action)
→
Middleware 1
→
Middleware 2
→
Reducer
→
New State

Firma de un middleware

Todo middleware Redux sigue la firma store => next => action => .... Esta estructura currificada permite encadenar múltiples middlewares.

JavaScript — middleware.js 
// ─── Middleware personalizado ─────────────────────────────
// Un middleware sigue la firma currificada:
//   store → next → action → resultado
// "next" llama al siguiente middleware (o al reducer si es el último).
// Si no llamamos next(action), la acción NO llega al reducer.

const miMiddleware = store => next => action => {

  // 1. Código ANTES de que la acción llegue al reducer
  console.log('Acción recibida:', action.type);
  console.log('Estado previo:', store.getState());

  // 2. Pasamos la acción al siguiente middleware/reducer
  const resultado = next(action);

  // 3. Código DESPUÉS de que el reducer actualizó el estado
  console.log('Estado nuevo:', store.getState());

  return resultado;
};

// ─── Registrar middlewares en el store ────────────────────
// applyMiddleware compone todos los middlewares en una cadena.
import { createStore, applyMiddleware } from 'redux';

const store = createStore(
  rootReducer,
  applyMiddleware(miMiddleware) // se pueden encadenar: applyMiddleware(a, b, c)
);
💡 Nota: Con Redux Toolkit, usa configureStore({ middleware: (getDefault) => getDefault().concat(miMiddleware) }) para agregar middlewares personalizados sin perder los de defecto.
02

Redux-Logger

¿Qué es redux-logger?

redux-logger es un middleware de depuración que imprime en la consola cada acción junto con el estado anterior y el nuevo estado. Es una herramienta de desarrollo — nunca debe ir a producción.

Bash — Instalación 
npm install redux-logger
JavaScript — store.js con Logger 
// ─── redux-logger ─────────────────────────────────────────
// createLogger acepta opciones para personalizar la salida:
//   collapsed: true → muestra los logs agrupados y colapsados
//   diff: true       → muestra solo las propiedades que cambiaron
//   predicate        → filtra qué acciones se loguean
// IMPORTANTE: Solo activo en NODE_ENV !== 'production'

import { createStore, applyMiddleware } from 'redux';
import { createLogger } from 'redux-logger';

// Configuración del logger
const logger = createLogger({
  collapsed: true,    // colapsa los grupos por defecto
  diff:       true,    // muestra diff entre estado previo y nuevo
  predicate:  (_, action) => action.type !== '@@INIT',
});

// Solo agregar el logger en desarrollo
const middlewares = [];
if (process.env.NODE_ENV !== 'production') {
  middlewares.push(logger);
}

const store = createStore(
  rootReducer,
  applyMiddleware(...middlewares)
);

Redux-Thunk

¿Qué es un thunk?

redux-thunk permite que dispatch acepte funciones además de objetos. La función recibida se ejecuta con (dispatch, getState) como argumentos, habilitando llamadas asíncronas, despachos condicionales y efectos secundarios.

Bash — Instalación 
npm install redux-thunk
# Con Redux Toolkit ya viene incluido por defecto
JavaScript — thunkActions.js 
// ─── Action Creator asíncrono con Thunk ───────────────────
// fetchUsuario NO retorna un objeto plano, sino una función.
// Redux-Thunk detecta que es función → la invoca con dispatch.
// Patrón típico: loading → éxito/error → fin.

import axios from 'axios';

export const fetchUsuario = (id) =>
  async (dispatch, getState) => {

    // Evitar petición duplicada si ya está cargando
    const { cargando } = getState().usuarios;
    if (cargando) return;

    dispatch({ type: 'FETCH_USUARIO_REQUEST' });

    try {
      const { data } = await axios.get(`/api/usuarios/${id}`);
      dispatch({ type: 'FETCH_USUARIO_SUCCESS', payload: data });
    } catch (error) {
      dispatch({
        type: 'FETCH_USUARIO_FAILURE',
        payload: error.message
      });
    }
  };

// ─── Uso en un componente React ───────────────────────────
import { useDispatch } from 'react-redux';

function Perfil() {
  const dispatch = useDispatch();
  // dispatch acepta la función gracias a redux-thunk
  dispatch(fetchUsuario(42));
}

Redux-Sagas

¿Qué es Redux-Saga?

redux-saga gestiona efectos secundarios complejos usando generadores (function*). A diferencia de Thunk, las sagas son funciones declarativas que describen lo que debe suceder, facilitando cancelaciones, reintentos, debouncing y flujos de concurrencia.

Característica Redux-Thunk Redux-Saga
Sintaxis async/await generators
Curva de aprendizaje Baja Alta
Cancelación de tareas Manual (AbortController) nativa (cancel)
Flujos complejos Difícil excelente
Testing Requiere mocks sin mocks (efectos)
Tamaño bundle ~1KB ~25KB
Bash — Instalación 
npm install redux-saga
JavaScript — sagas/usuarioSaga.js 
// ─── Redux-Saga: gestión asíncrona con generadores ────────
// Los efectos (call, put, takeLatest) son objetos planos que
// describen qué hacer — la librería los ejecuta.
//   call()        → llama una función asíncrona de forma testeable
//   put()         → despacha una acción al store
//   takeLatest()  → cancela sagas anteriores si llega nueva acción

import { call, put, takeLatest } from 'redux-saga/effects';
import axios from 'axios';

// Worker saga: ejecuta la lógica asíncrona
function* fetchUsuarioSaga({ payload: id }) {
  try {
    // call() crea un efecto testeable (no llama axios directamente)
    const { data } = yield call(axios.get, `/api/usuarios/${id}`);

    // put() despacha la acción de éxito
    yield put({ type: 'FETCH_USUARIO_SUCCESS', payload: data });
  } catch (error) {
    yield put({ type: 'FETCH_USUARIO_FAILURE', payload: error.message });
  }
}

// Watcher saga: escucha acciones y lanza workers
// takeLatest cancela el worker anterior si se despacha de nuevo
export function* usuarioWatcher() {
  yield takeLatest('FETCH_USUARIO_REQUEST', fetchUsuarioSaga);
}

// ─── Root saga y configuración del store ──────────────────
import { all } from 'redux-saga/effects';

export function* rootSaga() {
  yield all([
    usuarioWatcher(),
    // ...otras sagas
  ]);
}

// store.js
import createSagaMiddleware from 'redux-saga';

const sagaMiddleware = createSagaMiddleware();
const store = createStore(rootReducer, applyMiddleware(sagaMiddleware));

// Iniciar sagas DESPUÉS de crear el store
sagaMiddleware.run(rootSaga);
05

Deploy a Producción

Consideraciones para producción

Al llevar una app Redux a producción, es fundamental desactivar middlewares de desarrollo, optimizar el bundle y configurar correctamente las variables de entorno. El flag NODE_ENV=production controla este comportamiento.

1

Separar middlewares por entorno

Usar process.env.NODE_ENV !== 'production' para incluir redux-logger solo en desarrollo. Nunca expongas logs de estado en producción.

2

Configurar el store por entorno

Crear dos archivos: configureStore.dev.js y configureStore.prod.js, importando el correcto según NODE_ENV.

3

Build de producción

Ejecutar npm run build (Vite/CRA) genera un bundle minificado con dead code elimination, eliminando código de desarrollo.

4

Variables de entorno

Usar archivos .env.production para URLs de API. Con Vite: VITE_API_URL; con CRA: REACT_APP_API_URL.

5

Despliegue

Subir el contenido de /dist (Vite) o /build (CRA) a Vercel, Netlify, AWS S3 + CloudFront, o un servidor Nginx.

JavaScript — configureStore.js 
// ─── Store dinámico según NODE_ENV ───────────────────────
// En desarrollo: incluye logger + Redux DevTools Extension
// En producción: solo middlewares necesarios (thunk/saga)
// Este patrón mantiene el bundle de producción limpio.

import { configureStore } from '@reduxjs/toolkit';
import createSagaMiddleware from 'redux-saga';
import { rootSaga } from './sagas';
import { rootReducer } from './reducers';

const isDev = process.env.NODE_ENV !== 'production';

export function setupStore() {
  const sagaMiddleware = createSagaMiddleware();

  const store = configureStore({
    reducer: rootReducer,
    middleware: (getDefaultMiddleware) => {
      let chain = getDefaultMiddleware({ thunk: true })
        .concat(sagaMiddleware);

      // Logger SOLO en desarrollo (ver-sección-02)
      if (isDev) {
        const { createLogger } = require('redux-logger');
        chain = chain.concat(createLogger({ collapsed: true }));
      }

      return chain;
    },
    // Redux DevTools desactivado en producción automáticamente
    devTools: isDev,
  });

  sagaMiddleware.run(rootSaga);
  return store;
}

// ─── Uso en index.js / main.jsx ───────────────────────────
import { Provider } from 'react-redux';
import { setupStore } from './store/configureStore';

const store = setupStore();

ReactDOM.render(
  <Provider store={store}>
    <App />
  </Provider>,
  document.getElementById('root')
);
Bash — Build y deploy a Netlify/Vercel 
# 1. Build de producción (minifica y elimina código dev)
npm run build

# 2. Preview local del build de producción
npm run preview

# 3. Deploy a Netlify (CLI)
npx netlify-cli deploy --prod --dir=dist

# 4. Deploy a Vercel
npx vercel --prod
